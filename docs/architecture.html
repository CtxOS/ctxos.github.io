CTX OS Architecture

The Core Philosophy

CTX OS operates as a Configuration Layer. It does not replace the Debian base; it optimizes it. By using a modular postinst logic and custom /etc/ includes, we ensure that system updates from Debian never overwrite CTX OS optimizations.

1. The Apt Pipeline (Repository Layer)

CTX OS uses a "Single Source of Truth" model. The repository is hosted as a static site on GitHub Pages, but it is cryptographically secured.

  • Integrity: Every package is signed with a 4096-bit GPG key.
  • Availability: By leveraging GitHub's CDN, the OS can be deployed globally with zero downtime.
  • Automation: New versions are injected via reprepro, which updates the Packages.gz and InRelease files automatically.

2. Database & Resource Tuning

Instead of modifying the main postgresql.conf, CTX OS uses an Include Directory strategy.

  • The Logic: We inject a line into the main config: include = 'ctx_postgresql.conf'.
  • The Optimization: We specifically tune shared_buffers, effective_cache_size, and work_mem based on modern workstation standards rather than the conservative Debian defaults. This ensures that developer environments handle heavy queries without manual tuning.

3. Hardware & Security (Udev)

One of the most powerful features of CTX OS is the seamless hardware integration.

  • U2F/FIDO Security: We ship custom udev rules in /etc/udev/rules.d/. This allows non-root users to interact with security keys (like HyperFIDO or Yubikey) without needing sudo.
  • Device Mapping: When a recognized device is plugged in, the system automatically assigns the uaccess tag, enabling instant web-browser authentication for the logged-in user.
